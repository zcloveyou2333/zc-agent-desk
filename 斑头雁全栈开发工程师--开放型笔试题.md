# AI 工具重度使用与核心流程开发开放题

你好，感谢你参与我们的全栈开发岗位面试流程。

本次笔试只有 1 道开放题。我们希望了解你是否是 AI 工具的深度使用者，以及你如何重度运用 AI 工具完成需求拆解、判断取舍、核心流程设计和可运行项目交付。

本题不限制你使用任何第三方 AI 工具，也不限定具体投入时长。请在收到笔试作业题后 3 天内完成并提交；如有特殊情况，可以提前和 HR 沟通协商提交时间。你可以使用 Cursor、Claude Code、Codex、ChatGPT、Claude、Gemini、DeepSeek、通义千问、豆包、Copilot、Trae 或其他任意 AI 工具。

我们关注的不是 UI 是否精美，而是你是否能用 AI 工具把一个开放问题拆解清楚，并交付一个核心流程可运行、边界清楚、取舍明确的 MVP。

## 基本要求

- 题目数量：1 道开放题
- 时间安排：不限定具体投入时长；请在收到笔试作业题后 3 天内完成并提交，如有特殊情况，可以提前和 HR 沟通协商提交时间
- 必须使用 AI 工具完成项目
- 必须提交 GitHub 仓库，仓库中需要包含可运行代码
- 需要提交一段录屏或视频，用于展示你的解题思路和 AI 工具使用过程
- 录屏可以剪辑，可以隐藏隐私信息，不要求完整连续记录，不要求露脸
- 不限制技术栈
- 不限制使用的 AI 工具
- 不要提交真实 API Key、Token、个人账号信息或其他敏感信息

## 开放题

请使用 AI 工具完成一个可运行的 AI Chatbot MVP。

这个 Chatbot 面向内部员工使用。用户可以在网页中与 AI 多轮对话。系统不仅可以回答自然语言问题，还需要在合适的时候调用工具完成任务，例如查询时间、计算数据、创建待办、查询 mock 业务信息或调用你设计的本地工具。

你需要交付一个你认为短期内可以落地验证的 MVP。请重点保证核心流程跑通，而不是追求完整产品形态。

## 必须完成的核心流程

请至少跑通以下流程：

1. 用户可以在网页中发起一轮对话。
2. 系统可以返回 AI 回复。
3. 系统支持多轮上下文，用户连续追问时能保留前文信息。
4. 系统至少支持 1 个工具调用场景。
5. 工具调用需要由 AI 或工具路由逻辑判断触发，而不是只靠用户点击按钮。
6. 工具调用结果需要进入最终回复。
7. 系统需要有 mock 模式，在没有真实 LLM API Key 的情况下也能体验核心流程。
8. 项目可以根据 README 在本地启动运行。

## 建议实现范围

你可以自行决定技术栈和实现深度。建议优先完成：

- 一个简单可用的聊天页面
- 一个发送消息的后端接口
- 基础会话上下文管理
- 一个本地 mock LLM 或 mock provider
- 一个本地工具，例如时间查询、计算器、待办创建、mock 业务查询
- 工具参数校验和失败处理
- README 启动说明

UI 只要求清晰可用，不作为重点考察项。

## 需要在 README 或方案说明中写清楚

请在仓库 README 或单独方案文档中说明：

1. 你对需求的理解。
2. 你如何拆解 MVP 范围。
3. 第一版做了什么，明确不做什么。
4. 你为什么这样取舍。
5. 核心流程如何运行。
6. 前端、后端、模型、工具、存储之间的关系。
7. 多轮上下文如何处理。
8. 工具调用如何判断、执行和返回结果。
9. mock 模式如何启用。
10. 当前实现的限制和后续优化方向。
11. 你使用了哪些 AI 工具，以及分别用于哪些环节。
12. AI 工具输出中，你采纳了什么、修改了什么、拒绝了什么，为什么。

## 录屏要求

录屏用于展示你的解题思路和 AI 工具使用深度，不要求完整记录从 0 到 1 的所有开发过程。

录屏建议覆盖：

1. 你如何理解题目和定义 MVP。
2. 你如何向 AI 工具提问、拆解任务和推进实现。
3. AI 给出建议或代码后，你如何判断、追问、修改或拒绝。
4. 你如何围绕核心流程做取舍。
5. 你如何验证项目可运行。
6. 你如何演示普通对话、多轮追问、工具调用和 mock 模式。
7. 你认为当前实现最大的限制是什么，后续会如何优化。

录屏可以剪辑，可以快进等待时间，可以隐藏隐私信息。重点是让评审看到你的判断过程，而不是录制完整操作时长。

## 交付物

请提交：

1. GitHub 仓库链接。
2. 录屏或视频链接。
3. 如 README 中没有完整说明，请额外提交一份简短方案说明。

GitHub 仓库需要包含：

- 完整代码
- README
- 本地启动方式
- 环境变量说明
- mock 模式说明
- 工具调用说明
- 不包含真实 API Key 或敏感信息

## 评分维度

我们会重点关注以下方面：

| 维度 | 权重 | 关注点 |
| --- | ---: | --- |
| AI 工具重度使用能力 | 30% | 是否能高质量使用 AI 工具完成拆解、实现、调试和复盘；是否能判断、追问、修正和拒绝 AI 输出 |
| 需求拆解与取舍 | 25% | 是否能从开放需求中拆出核心流程、MVP 边界、优先级和不做事项 |
| 核心流程可运行性 | 25% | 普通对话、多轮上下文、工具调用、mock 模式、本地启动是否真正跑通 |
| 技术方案与工程意识 | 15% | 前后端边界、模型调用、工具调用、错误处理、参数校验、API Key 安全是否清楚 |
| 表达清晰度 | 5% | README、录屏和方案说明是否能让评审快速理解你的判断和实现 |

## 不重点考察的内容

以下内容不是本次笔试重点：

1. UI 视觉设计是否精美。
2. 是否使用某个指定 AI 工具。
3. 是否使用某个指定技术栈。
4. 是否部署到线上。
5. 是否支持复杂权限系统。
6. 是否支持复杂知识库。
7. 是否支持多工具编排。
8. 是否达到生产级稳定性。

如果你实现了这些能力，可以作为加分说明，但请优先保证核心流程清晰、可运行、可解释。

## 作答建议

请把时间优先投入在以下问题上：

1. 这个 MVP 到底解决什么问题？
2. 哪些是第一版必须跑通的核心流程？
3. 哪些功能应该暂时放弃？
4. AI 工具在哪些环节真正提升了你的效率和方案质量？
5. AI 工具输出中有哪些地方需要人工判断和修正？
6. 你的项目如何让评审快速启动、体验和理解？

我们希望看到的是一个具有深度 AI 工具使用习惯的开发者，如何在开放问题中快速拆解、判断、实现和验证，而不是简单堆功能或复制 AI 输出。
